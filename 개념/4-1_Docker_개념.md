# 4️⃣-1️⃣ Docker 개념

---

## 1️⃣ 이미지(Image) vs 컨테이너(Container)

---

### 📦 비유: 레시피 책 vs 실제 요리

| 구분 | 비유 | 의미 |
| --- | --- | --- |
| 이미지 | 레시피 | 실행 준비된 설계도 |
| 컨테이너 | 요리 | 실제 실행 중인 것 |

---

### 💻 기술적 정의

### 이미지(Image)

- 실행 환경이 정의된 **불변 파일**
- OS 일부 + 라이브러리 + 앱

### 컨테이너(Container)

- 이미지를 실행한 상태
- 메모리·CPU 사용
- 프로세스 실행 중

---

### 📌 예시

```bash
docker pull nginx
docker run nginx

```

---

### 🔍 이 프로젝트에서

- nginx 이미지
- 실행된 컨테이너 = 웹 서버

👉 **이미지는 배포, 컨테이너는 운영**

---

## 2️⃣ 컨테이너는 왜 가벼운가?

---

### 🏠 비유: 독립 주택 vs 원룸 쉐어

---

### 🧱 VM 방식

- 집마다:
    - 건물
    - 전기
    - 수도

→ 무겁고 느림

---

### 📦 컨테이너 방식

- 건물은 공유
- 방만 나눔

→ 가볍고 빠름

---

### 💻 기술적 이유

> 컨테이너는 호스트 OS 커널을 공유
> 
- 별도 OS ❌
- 필요한 것만 포함

---

### ⚡ 결과

- 빠른 실행
- 적은 자원
- 높은 밀도

---

### 🔍 이 프로젝트에서

- 하나의 EC2에
- 여러 서비스 실행

👉 **비용 절감 + 효율 극대화**

---

## 3️⃣ 컨테이너 생명주기

---

### 🔁 비유: 전구의 수명

1. 만들어짐
2. 켜짐
3. 꺼짐
4. 교체됨

---

### 💻 실제 흐름

```
이미지 → 생성(create) → 실행(run)
   ↓
중지(stop) → 삭제(remove)

```

---

### ⚠️ 중요한 특징

- 컨테이너는 **일시적**
- 꺼지면 내부 데이터 사라짐

---

### 🔍 이 프로젝트에서

- 데이터는 볼륨에 저장
- 컨테이너는 언제든 교체 가능

👉 **이 개념이 백업의 출발점**

---

# 🧠 4-1단계 핵심 요약

- 이미지는 설계도, 컨테이너는 실행체
- 컨테이너는 OS를 공유해 매우 가볍다
- 컨테이너는 생성·실행·삭제되는 일회성 존재

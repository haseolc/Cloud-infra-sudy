# 2️⃣-1️⃣ IaC 개념

## 1️⃣ “코드로 인프라를 정의한다”는 의미

---

### 🏗️ 비유: 건축 도면

- 건물을 직접 벽돌 쌓는 것 ❌
- **도면을 그려서** 시공사가 그대로 만듦 ✅

👉 도면 = 코드

👉 건물 = 서버 / 네트워크 / 보안 설정

---

### 💻 기술적 정의

> IaC = 서버, 네트워크, 보안 설정을 ‘코드’로 작성해 자동 생성하는 방식
> 
- 서버 개수
- OS 종류
- 포트 개방
- 디스크 크기

전부 **파일에 명시**

---

### ❌ IaC가 없으면

- “이 서버 어떻게 만들었지?”
- “저번엔 됐는데 이번엔 왜 안 돼?”

---

### 🔍 이 프로젝트에서

- Terraform으로:
    - EC2
    - 보안 그룹
    - 키 페어
    - 네트워크

👉 **수동 클릭 0%**

---

## 2️⃣ 재현 가능성 (Reproducibility)

---

### 🔁 비유: 공장 생산라인

- 수작업 → 제품마다 품질 다름
- 자동화 → **항상 동일**

---

### 💻 인프라에서 재현성이란?

> 언제, 어디서, 누가 실행해도 동일한 인프라가 생성되는 것
> 
- 개발자 A
- 개발자 B
- CI 서버

👉 결과 동일

---

### 🧠 왜 중요한가?

- 팀 협업
- 테스트 환경 = 운영 환경
- 장애 재현 가능

---

### 🔍 이 프로젝트에서

- 팀원이 Terraform 실행
- 동일한 AWS 인프라 생성
- 실습·발표·재현 가능

👉 **프로젝트 평가에서 큰 가산점**

---

## 3️⃣ 버전 관리 (Version Control)

---

### 📚 비유: 설계도 이력 관리

- 1층 → 2층 → 3층
- 언제든 이전 버전으로 롤백

---

### 💻 기술적 의미

> 인프라 변경 이력을 Git으로 관리
> 
- 누가
- 언제
- 무엇을

변경했는지 추적 가능

---

### 🧠 코드이기 때문에 가능한 것

- GitHub에 저장
- PR 리뷰
- 변경 이력 비교(diff)

---

### 🔥 장애 발생 시

- “어제 뭐 바뀌었지?”
- → Git 로그 확인
- → 이전 상태로 복구

---

### 🔍 이 프로젝트에서

- Terraform 코드 = GitHub
- GitHub Actions와 연계
- 자동 배포 파이프라인 구성

👉 **운영 자동화의 핵심 연결고리**

---

# 🧠 2-1단계 핵심 요약

- IaC는 인프라를 코드로 관리하는 개념
- 재현 가능성은 자동화의 핵심 가치
- 코드이기 때문에 버전 관리와 롤백이 가능

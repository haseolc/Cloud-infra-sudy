# 5️⃣-3️⃣ 운영 개념

## 1️⃣ Scaling (확장)

---

### 📈 비유: 알바 인원 늘리기

- 손님 많으면 사람 더 뽑음
- 손님 줄면 다시 줄임

---

### 💻 기술적 정의

> Scaling = Pod 개수를 조절하여 처리량을 조정하는 것
> 

---

### 🔧 방식

- **수동 Scaling**
    
    ```bash
    kubectl scale deployment web --replicas=5
    
    ```
    
- **자동 Scaling (HPA)**
    - CPU / 메모리 기준
    - 요청 수 기준

---

### 🔍 이 프로젝트에서

- 트래픽 증가 → Pod 자동 증가
- 비용 최적화 가능

👉 **가용성과 비용의 균형**

---

## 2️⃣ Self-healing (자가 치유)

---

### 🩹 비유: 쓰러지면 자동 교체

- 직원 쓰러지면
- 바로 대체 인원 투입

---

### 💻 기술적 정의

> Self-healing = 장애 발생 시 자동으로 상태를 복구하는 기능
> 

---

### 🧠 내부 동작

- Pod 죽음 감지
- Replica 수 확인
- 즉시 새 Pod 생성

---

### 🔍 이 프로젝트에서

- 컨테이너 크래시 → 자동 복구
- 운영자 개입 최소화

👉 **24시간 무중단 운영 핵심**

---

## 3️⃣ Rolling Update

---

### 🔄 비유: 가게 공사하면서 영업

- 문 닫지 않고
- 하나씩 교체

---

### 💻 기술적 정의

> Rolling Update = 서비스 중단 없이 버전을 교체하는 방식
> 

---

### ⚙️ 작동 방식

- 기존 Pod 일부 종료
- 새 Pod 기동
- 점진적 교체

---

### 🔧 Deployment 전략

```yaml
strategy:
type:RollingUpdate

```

---

### 🔍 이 프로젝트에서

- Docker 이미지 버전 업데이트
- 사용자 체감 다운타임 0

👉 **실무에서 가장 많이 쓰는 배포 방식**

---

## 4️⃣ Rollback

---

### ⏪ 비유: 문제 생기면 원상 복구

- 신메뉴 실패
- 이전 메뉴로 즉시 복귀

---

### 💻 기술적 정의

> Rollback = 이전 안정 버전으로 되돌리는 기능
> 

---

### 🔧 명령 예시

```bash
kubectl rollout undo deployment web

```

---

### 🧠 이유

- 배포는 항상 위험
- 빠른 복구가 생명

---

### 🔍 이 프로젝트에서

- 장애 발생 시 즉시 복구
- 서비스 신뢰도 확보

👉 **운영자의 생존 버튼**

---

# 🧠 5-3단계 핵심 요약

- Scaling → 처리량 조절
- Self-healing → 자동 장애 복구
- Rolling Update → 무중단 배포
- Rollback → 빠른 안정 복구

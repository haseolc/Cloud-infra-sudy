# 0️⃣ 전체 큰 그림 – 서비스 운영의 본질

## 1️⃣ 서비스 운영이란 무엇인가?

### 📌 한 줄 정의

> 서비스 운영 = 사용자가 “항상 문제없이” 서비스를 사용할 수 있게 유지하는 일
> 

---

### 🏪 비유: 무인 편의점

- 편의점이 하루만 열고 끝나면? → **운영 아님**
- 매일 열리고, 물건이 채워지고, 고장 나면 복구됨 → **운영**

👉 웹 서비스도 동일해.

---

### 💻 서비스 운영에서 항상 요구되는 것

| 요소 | 의미 |
| --- | --- |
| 가용성 | 서비스가 항상 켜져 있어야 함 |
| 안정성 | 갑자기 죽지 않아야 함 |
| 확장성 | 사람이 많아지면 버텨야 함 |
| 보안 | 외부 공격을 막아야 함 |
| 복구 | 망가져도 다시 살아나야 함 |

---

### 🔍 이 프로젝트에서

단순히 “웹 서버 하나 띄우기”가 아니라:

- 자동으로 켜지고
- 공격에 대비하고
- 문제가 나면 복구되고
- 데이터가 사라지지 않는

👉 **“운영 가능한 서비스”를 만드는 것**

---

## 2️⃣ 수동 구축 vs 자동화 구축

---

### 🧑‍🔧 수동 구축 (사람 손)

### 비유: 요리 레시피 없이 감으로 요리

- 서버 하나 만들 때마다:
    - 클릭
    - 명령어 입력
    - 기억에 의존
- 다음에 다시 하려면?
    - “어… 이거 전에 어떻게 했지?”

### 문제점

- 재현 불가능
- 실수 발생
- 팀원 간 환경 불일치

---

### 🤖 자동화 구축 (코드)

### 비유: 정확한 레시피 + 자동 조리기

- 재료(코드)만 있으면
- 언제 어디서나
- 똑같은 결과

---

### 🧠 핵심 차이

| 항목 | 수동 | 자동화 |
| --- | --- | --- |
| 재현성 | ❌ | ✅ |
| 실수 | 많음 | 거의 없음 |
| 팀 협업 | 어려움 | 쉬움 |
| 운영 | 불가능 | 가능 |

---

### 🔍 이 프로젝트에서

- Terraform → 서버 자동 생성
- Kubernetes YAML → 컨테이너 자동 관리
- GitHub Actions → 자동 배포

👉 **“사람이 하지 않는 운영”이 목표**

---

## 3️⃣ Desired State (원하는 상태) 개념

> ⭐ 이 개념은 Kubernetes의 핵심 사상
> 

---

### 📌 정의

> Desired State = 시스템이 ‘항상 이렇게 있어야 한다’고 정의한 상태
> 

---

### 🚗 비유: 자동차 크루즈 컨트롤

- 설정: “속도 100km 유지”
- 언덕 올라가면?
    - 시스템이 자동으로 가속
- 내리막이면?
    - 자동 감속

👉 **중요한 건 ‘어떻게’가 아니라 ‘어디에 있어야 하는지’**

---

### 💻 Kubernetes에서

```yaml
replicas:3

```

의미:

- “나는 항상 Pod 3개가 필요해”
- 하나 죽으면?
    - Kubernetes가 자동으로 다시 생성

---

### 🔍 이 프로젝트에서

- 사람이 “살려라”라고 명령하지 않아도
- 시스템이 **스스로 상태를 복구**

👉 이게 바로 **Self-healing**

---

## 4️⃣ 선언형 설정 vs 명령형 실행

---

### 🧾 명령형 (Imperative)

### 비유: 요리 하나하나 지시

- “불 켜”
- “고기 올려”
- “3분 뒤집어”

```bash
kubectl run ...
kubectl expose ...

```

❌ 사람이 계속 개입해야 함

---

### 📜 선언형 (Declarative)

### 비유: 주문서 한 장

- “스테이크 미디엄 레어 하나 주세요”

```yaml
replicas:3
image:nginx

```

👉 **시스템이 알아서 수행**

---

### 🧠 차이 요약

| 구분 | 명령형 | 선언형 |
| --- | --- | --- |
| 초점 | 과정 | 결과 |
| 자동 복구 | ❌ | ✅ |
| 운영 적합 | ❌ | ✅ |

---

### 🔍 이 프로젝트에서

- Terraform → 선언형 인프라
- Kubernetes YAML → 선언형 서비스 운영

👉 운영 가능한 구조의 핵심

---

## 5️⃣ 장애란 무엇이고, 왜 대비해야 하는가?

---

### 💥 장애란?

> “정상 상태가 깨진 모든 상황”
> 
- 서버 다운
- 컨테이너 죽음
- 네트워크 끊김
- 데이터 삭제
- 해킹

---

### 🧯 비유: 아파트 화재 대비

- “불 안 나면 좋지” ❌
- “불은 반드시 난다” ✅
- 중요한 건 **대응 능력**

---

### 🔥 장애는 피할 수 없다

- 하드웨어는 고장남
- 소프트웨어는 버그 있음
- 사람은 실수함

👉 **그래서 대비가 ‘옵션’이 아니라 ‘전제’**

---

### 🛡️ 이 프로젝트에서의 대비

- Kubernetes → 자동 복구
- ReplicaSet → 개수 유지
- 백업 → 데이터 복구
- 모니터링 → 조기 감지

---

## 1️⃣ 개발 환경과 인프라 환경 구분

### 🔹 보완 정리

- **개발 환경(Local)**: 팀원 개인 PC (Windows)
- **운영 환경(Cloud)**: AWS EC2 상의 Ubuntu Server

### ✔️ 정리 이유

- 개인 PC OS는 **프로젝트 핵심 설계 요소가 아님**
- 설계 문서에서는 *운영 환경* 중심으로 기술하는 것이 적절

---

## 2️⃣ Terraform의 역할 명확화

### 🔹 보완 정리

- **인프라 정의 도구 (IaC, Infrastructure as Code)**: Terraform

### ✔️ 설명 문장 (바로 사용 가능)

> Terraform은 서버를 직접 가상화 하는 도구가 아니라,
> 
> 
> 클라우드 인프라(AWS EC2, 네트워크, 보안 그룹 등)를
> 
> **코드로 정의하고 자동 생성·관리하기 위한 IaC 도구**이다.
> 

---

## 3️⃣ 인프라 자동화 단계 표현 정리

### 🔹 기존 표현

- 가상화 도구: Terraform
- 서버 OS: Ubuntu
- 서버 위치: AWS EC2

### 🔹 보완된 단계 정리

1. **Terraform을 이용한 AWS 인프라 정의**
    - EC2 인스턴스
    - 보안 그룹
    - 네트워크 설정
2. **Ubuntu Server 자동 구축**
3. **Docker 설치 및 컨테이너 실행 환경 구성**
4. **Kubernetes 클러스터 구성**

### ✔️ 정리 이유

- “가상화”라는 모호한 표현 제거
- **인프라 → OS → 컨테이너 → 오케스트레이션** 흐름 명확화

---

## 4️⃣ Kubernetes 역할 표현 정리

### 🔹 기존 표현

- YAML 해석 도구: Kubernetes

### 🔹 보완 정리

- **컨테이너 오케스트레이션 도구**: Kubernetes

### ✔️ 설명 문장

> Kubernetes는 YAML 파일을 단순히 해석하는 도구가 아니라,
> 
> 
> 정의된 상태(desired state)를 기준으로
> 
> 컨테이너의 배포, 확장, 복구를 자동으로 관리하는 오케스트레이션 플랫폼이다.
> 

---

## 5️⃣ GitHub Actions의 위치 명확화

### 🔹 기존 표현

- YAML 파일 생성: GitHub Actions를 통한 Kubernetes 배포 자동화

### 🔹 보완 정리

- **CI/CD 도구**: GitHub Actions
- 역할:
    - Terraform 실행 자동화
    - Kubernetes YAML 배포 자동화

### ✔️ 설명 문장

> GitHub Actions를 통해 코드 변경 시
> 
> 
> 인프라 구성(Terraform)과 애플리케이션 배포(Kubernetes)를
> 
> 자동으로 수행하는 CI/CD 파이프라인을 구성한다.
> 

---

## 6️⃣ 정리된 기술 스택 한눈 요약

| 구분 | 기술 | 역할 |
| --- | --- | --- |
| IaC | Terraform | 클라우드 인프라 자동 생성 |
| OS | Ubuntu Server | 컨테이너 실행 환경 |
| Container | Docker | 애플리케이션 패키징 |
| Orchestration | Kubernetes | 배포·확장·복구 자동화 |
| CI/CD | GitHub Actions | 자동 배포 파이프라인 |

## 1️⃣ 서비스 운영이란 무엇인가?

### 📌 한 줄 정의

> 서비스 운영 = 사용자가 “항상 문제없이” 서비스를 사용할 수 있게 유지하는 일
> 

---

### 🏪 비유: 무인 편의점

- 편의점이 하루만 열고 끝나면? → **운영 아님**
- 매일 열리고, 물건이 채워지고, 고장 나면 복구됨 → **운영**

👉 웹 서비스도 동일해.

---

### 💻 서비스 운영에서 항상 요구되는 것

| 요소 | 의미 |
| --- | --- |
| 가용성 | 서비스가 항상 켜져 있어야 함 |
| 안정성 | 갑자기 죽지 않아야 함 |
| 확장성 | 사람이 많아지면 버텨야 함 |
| 보안 | 외부 공격을 막아야 함 |
| 복구 | 망가져도 다시 살아나야 함 |

---

### 🔍 이 프로젝트에서

단순히 “웹 서버 하나 띄우기”가 아니라:

- 자동으로 켜지고
- 공격에 대비하고
- 문제가 나면 복구되고
- 데이터가 사라지지 않는

👉 **“운영 가능한 서비스”를 만드는 것**

---

## 2️⃣ 수동 구축 vs 자동화 구축

---

### 🧑‍🔧 수동 구축 (사람 손)

### 비유: 요리 레시피 없이 감으로 요리

- 서버 하나 만들 때마다:
    - 클릭
    - 명령어 입력
    - 기억에 의존
- 다음에 다시 하려면?
    - “어… 이거 전에 어떻게 했지?”

### 문제점

- 재현 불가능
- 실수 발생
- 팀원 간 환경 불일치

---

### 🤖 자동화 구축 (코드)

### 비유: 정확한 레시피 + 자동 조리기

- 재료(코드)만 있으면
- 언제 어디서나
- 똑같은 결과

---

### 🧠 핵심 차이

| 항목 | 수동 | 자동화 |
| --- | --- | --- |
| 재현성 | ❌ | ✅ |
| 실수 | 많음 | 거의 없음 |
| 팀 협업 | 어려움 | 쉬움 |
| 운영 | 불가능 | 가능 |

---

### 🔍 이 프로젝트에서

- Terraform → 서버 자동 생성
- Kubernetes YAML → 컨테이너 자동 관리
- GitHub Actions → 자동 배포

👉 **“사람이 하지 않는 운영”이 목표**

---

## 3️⃣ Desired State (원하는 상태) 개념

> ⭐ 이 개념은 Kubernetes의 핵심 사상
> 

---

### 📌 정의

> Desired State = 시스템이 ‘항상 이렇게 있어야 한다’고 정의한 상태
> 

---

### 🚗 비유: 자동차 크루즈 컨트롤

- 설정: “속도 100km 유지”
- 언덕 올라가면?
    - 시스템이 자동으로 가속
- 내리막이면?
    - 자동 감속

👉 **중요한 건 ‘어떻게’가 아니라 ‘어디에 있어야 하는지’**

---

### 💻 Kubernetes에서

```yaml
replicas:3

```

의미:

- “나는 항상 Pod 3개가 필요해”
- 하나 죽으면?
    - Kubernetes가 자동으로 다시 생성

---

### 🔍 이 프로젝트에서

- 사람이 “살려라”라고 명령하지 않아도
- 시스템이 **스스로 상태를 복구**

👉 이게 바로 **Self-healing**

---

## 4️⃣ 선언형 설정 vs 명령형 실행

---

### 🧾 명령형 (Imperative)

### 비유: 요리 하나하나 지시

- “불 켜”
- “고기 올려”
- “3분 뒤집어”

```bash
kubectl run ...
kubectl expose ...

```

❌ 사람이 계속 개입해야 함

---

### 📜 선언형 (Declarative)

### 비유: 주문서 한 장

- “스테이크 미디엄 레어 하나 주세요”

```yaml
replicas:3
image:nginx

```

👉 **시스템이 알아서 수행**

---

### 🧠 차이 요약

| 구분 | 명령형 | 선언형 |
| --- | --- | --- |
| 초점 | 과정 | 결과 |
| 자동 복구 | ❌ | ✅ |
| 운영 적합 | ❌ | ✅ |

---

### 🔍 이 프로젝트에서

- Terraform → 선언형 인프라
- Kubernetes YAML → 선언형 서비스 운영

👉 운영 가능한 구조의 핵심

---

## 5️⃣ 장애란 무엇이고, 왜 대비해야 하는가?

---

### 💥 장애란?

> “정상 상태가 깨진 모든 상황”
> 
- 서버 다운
- 컨테이너 죽음
- 네트워크 끊김
- 데이터 삭제
- 해킹

---

### 🧯 비유: 아파트 화재 대비

- “불 안 나면 좋지” ❌
- “불은 반드시 난다” ✅
- 중요한 건 **대응 능력**

---

### 🔥 장애는 피할 수 없다

- 하드웨어는 고장남
- 소프트웨어는 버그 있음
- 사람은 실수함

👉 **그래서 대비가 ‘옵션’이 아니라 ‘전제’**

---

### 🛡️ 이 프로젝트에서의 대비

- Kubernetes → 자동 복구
- ReplicaSet → 개수 유지
- 백업 → 데이터 복구
- 모니터링 → 조기 감지

## 1️⃣ 디렉토리 구조

(`/etc`, `/var`, `/home`)

---

### 🗄️ 비유: 회사 건물의 층별 역할

| 디렉토리 | 비유 | 역할 |
| --- | --- | --- |
| `/etc` | 운영 규칙 문서실 | 설정 파일 |
| `/var` | 물류 창고 | 로그·데이터 |
| `/home` | 개인 사무실 | 사용자 파일 |

---

### 💻 기술적 설명

### `/etc`

- 시스템 설정 파일
- nginx, ssh 설정

### `/var`

- 로그 (`/var/log`)
- DB 데이터
- 캐시

### `/home`

- 사용자 홈
- 작업 파일

---

### 🔍 이 프로젝트에서

- 백업 대상:
    - `/var/www`
    - `/var/lib/mysql`
- 보안 설정:
    - `/etc/nginx`
    - `/etc/ssh`

👉 **백업 범위 판단의 기준**

---

## 2️⃣ 사용자 / 권한

---

### 🔐 비유: 출입증과 직급

- 누구인가?
- 무엇을 할 수 있는가?

---

### 💻 개념

> Linux는 사용자 + 그룹 + 권한으로 모든 접근을 통제
> 

---

### 📄 권한 구조

```
rwxr-x---

```

- 읽기(r)
- 쓰기(w)
- 실행(x)

---

### 🔍 이 프로젝트에서

- root 최소 사용
- 서비스 계정 분리
- DB 접근 권한 제한

👉 **침해 사고 최소화**

---

## 3️⃣ 프로세스 개념

---

### 🧠 비유: 실제로 일하는 직원

- 프로그램 = 직원 명단
- 프로세스 = 지금 일 중인 직원

---

### 💻 기술적 정의

> 프로세스 = 실행 중인 프로그램 인스턴스
> 

---

### 📌 예시

- nginx 실행 → 여러 worker 프로세스
- mysql 데몬

---

### 🔍 이 프로젝트에서

- 죽은 프로세스 확인
- 장애 원인 분석
- Kubernetes의 Pod 개념과 연결

👉 **“서버가 살아 있는지” 판단 기준**

---

## 4️⃣ 포트 확인

---

### 🚪 비유: 문이 열려 있는지 확인

- 서버 주소 = 건물
- 포트 = 문 번호

---

### 💻 개념

> 포트는 프로세스가 외부와 통신하는 통로
> 

---

### 📌 확인 명령

```bash
ss -tulnp

```

---

### 🔍 이 프로젝트에서

- 80 / 443 → Nginx
- 내부 포트는 비공개

👉 **보안 점검 필수 단계**

---

## 5️⃣ 서비스 실행 방식

---

### 🔁 비유: 자동 출근 vs 수동 출근

---

### 🧑‍🔧 수동 실행

```bash
nginx

```

- 재부팅 시 꺼짐

---

### 🤖 서비스 등록

```bash
systemctlenable nginx

```

- 부팅 시 자동 실행

---

### 💻 개념

> Linux 서비스는 systemd가 관리
> 

---

### 🔍 이 프로젝트에서

- 서버 재부팅에도 서비스 유지
- 장애 복구 자동화

👉 **운영 서버의 기본 조건**

### 📜 비유: CCTV + 업무일지

- 문제 발생 시 원인 추적

### 💻 로그란?

> 시스템과 서비스의 행동 기록
> 

---

### 📂 주요 로그 위치

- `/var/log/syslog`
- `/var/log/nginx/access.log`
- `/var/log/nginx/error.log`

---

### 🔍 이 프로젝트에서

- 접속 기록 분석
- 공격 탐지
- 장애 원인 분석

### 🧠 비유

- **모니터링** → 심박수·체온 체크 (실시간 상태)
- **로깅** → 병원 진료 기록 (사후 분석)

---

### 📊 차이 정리

| 구분 | 모니터링 | 로깅 |
| --- | --- | --- |
| 목적 | 이상 징후 감지 | 원인 분석 |
| 시점 | 사전 / 실시간 | 사후 |
| 데이터 | 숫자(메트릭) | 텍스트 |
| 도구 | Prometheus | ELK, Loki |

---

### 🧠 핵심 정리

> 운영 = 모니터링 + 로깅
> 
> 
> 둘 중 하나만 있으면 반쪽짜리 운영
> 

---

## 2️⃣ Prometheus

---

### 🧠 한 줄 정의

> Prometheus = 메트릭 수집 서버
> 

---

### 💡 역할

- 서버, 컨테이너, Pod 상태 수집
- CPU, 메모리, 요청 수, 지연 시간 등

---

### 🧩 동작 방식 (중요)

> Pull 방식
> 

```
Prometheus
   ← metrics 제공 (HTTP)
   ← 서버 / 컨테이너 / K8s

```

---

### 🧠 왜 Pull 방식?

- 대상 장애 시 감지 쉬움
- 중앙 집중 관리

---

## 3️⃣ Grafana

---

### 🧠 한 줄 정의

> Grafana = 메트릭 시각화 도구
> 

---

### 💡 역할

- Prometheus 데이터 시각화
- 대시보드 구성
- 운영자 한 눈에 상태 파악

---

### 📊 예시 대시보드

- CPU 사용률
- 메모리 사용량
- 요청 수 / 에러율
- Pod 개수 변화

---

### 🧠 실무 포인트

> “Grafana는 의사 결정용 화면이다”
> 

---

## 4️⃣ 메트릭 (Metrics)

---

### 🧠 정의

> 시스템 상태를 숫자로 표현한 값
> 

---

### 📊 대표 메트릭

| 메트릭 | 의미 |
| --- | --- |
| CPU Usage | 부하 상태 |
| Memory Usage | OOM 위험 |
| Request Count | 트래픽 |
| Error Rate | 장애 징후 |
| Latency | 성능 저하 |

---

### 🧠 핵심

> 로그는 읽고,
> 
> 
> 메트릭은 **본다**
> 

---

## 5️⃣ 알람 개념

---

### 🧠 한 줄 정의

> 알람 = 기준 초과 시 울리는 경고
> 

---

### 📄 예시

- CPU 80% 이상 5분 지속
- 에러율 5% 초과
- Pod 재시작 횟수 급증

---

### 🧠 알람의 목적

- “문제 있다” 알리기
- 자동 대응 트리거

---

### ⚠️ 실무 경고

> 알람 많으면 → 다 무시됨 ❌
> 
> 
> **진짜 위험한 것만 울려야 함**
> 

---

## 6️⃣ 장애 감지 흐름

---

### 🧩 전체 흐름 다이어그램 (텍스트)

```
서비스 이상 발생
   ↓
메트릭 수치 변화
   ↓
Prometheus 수집
   ↓
알람 조건 만족
   ↓
Alert 발생
   ↓
운영자 대응 / 자동 조치

```

---

### 🧠 핵심

> 장애는 “발생”보다
> 
> 
> **“감지까지 걸린 시간”이 더 중요**
>

## 🔵 한 줄 요약 (면접/발표용)

> 장애는 언제든 발생한다는 전제로 설계해야 하며,
> 
> 
> 복구는 **자동 + 우선순위 기반**으로 이루어져야 한다.
> 

---

## 1️⃣ 컨테이너 장애

---

### 🧠 비유

- **컨테이너 = 음식점 직원**
- 직원 한 명 쓰러짐 ≠ 가게 폐업

---

### 📌 장애 상황

- 프로세스 크래시
- 애플리케이션 예외 종료
- 메모리 초과(OOM)

---

### 🔄 쿠버네티스 대응

- 컨테이너 재시작
- Pod 내부에서 자동 복구

---

### 🧠 핵심 포인트

> 컨테이너 장애는 “사소한 장애”
> 
> 
> → 자동 복구 전제
> 

---

## 2️⃣ Pod 장애

---

### 🧠 비유

- **Pod = 음식점 한 테이블**
- 테이블이 깨지면 새 테이블로 교체

---

### 📌 장애 상황

- Pod 삭제
- 노드와 통신 불가
- 리소스 부족

---

### 🔄 쿠버네티스 대응

- ReplicaSet이 새로운 Pod 생성
- Desired State 유지

---

### 🧠 핵심 포인트

> Pod는 언제든 죽어도 되는 존재
> 

---

## 3️⃣ 서버(Node) 장애

---

### 🧠 비유

- **서버 = 음식점 건물**
- 건물이 무너지면 다른 건물로 손님 이동

---

### 📌 장애 상황

- EC2 다운
- 네트워크 단절
- 디스크 오류

---

### 🔄 쿠버네티스 대응

- 해당 노드에서 Pod 제거
- 다른 노드에 재배치

---

### ⚠️ 전제 조건

- 노드가 **2대 이상**
- 스토리지는 외부화

---

### 🧠 핵심 포인트

> 서버 장애는 설계로 대비
> 

---

## 4️⃣ 데이터 손실

---

### 🧠 비유

- **데이터 = 영업 장부**
- 서버가 살아도 장부가 날아가면 끝

---

### 📌 장애 상황

- 컨테이너 삭제
- 볼륨 손상
- DB 파일 손실

---

### 🔄 대응 전략

| 항목 | 대응 |
| --- | --- |
| 파일 | rsync / tar 백업 |
| DB | dump 백업 |
| 전체 | 스냅샷 |

---

### 🧠 핵심 포인트

> 데이터는 절대 자동 복구 안 됨
> 
> 
> → 백업만이 답
> 

---

## 5️⃣ 복구 우선순위

---

### 🧠 비유

- 병원 응급실 트리아지(분류)

---

### 📊 복구 우선순위 표

| 우선순위 | 대상 | 이유 |
| --- | --- | --- |
| 1 | 서비스 접근성 | 사용자가 먼저 |
| 2 | 데이터 무결성 | 영구 손실 방지 |
| 3 | 애플리케이션 | 기능 복구 |
| 4 | 성능 최적화 | 이후 단계 |

---

### 🧠 핵심 문장

> “모든 걸 동시에 고치려다
> 
> 
> **아무 것도 못 고친다**”
> 

---

## 6️⃣ 전체 장애 → 복구 흐름

```
장애 발생
   ↓
모니터링 감지
   ↓
알람 발생
   ↓
자동 복구 (K8s)
   ↓
백업 복구 (필요 시)
   ↓
정상화

```
